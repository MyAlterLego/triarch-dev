Step 1 · Choose how to analyze

Both paths produce the same Triarch-branded findings format. The form is faster but coarser; the discovery script is more precise.

Answer 10 questions

Best when you already know your repo's state and just want a fast assessment.

~3 minutes · runs entirely in your browser

Path B · Discovery + Claude Code

Four short steps. Nothing leaves your laptop.

0. Authenticate to GitHub + your cloud (one-time)

These are interactive logins — they open a browser, you sign in, the CLI stores a session. Triarch never sees any of these tokens. Skip whatever clouds you don't use.

GitHub — required

# Check current state — if already logged in with the right scopes, skip the login step
gh auth status

# If not authenticated, login (browser flow):
gh auth login --hostname github.com --git-protocol https \
  --scopes "repo,read:org,workflow"

AWS — if you use AWS

# Check first
aws sts get-caller-identity

# If not authenticated, pick ONE of these:

# Option A — AWS SSO / IAM Identity Center (recommended for org-managed accounts)
aws configure sso         # first time only
aws sso login             # every ~8 hours

# Option B — federated via your IdP (Okta / Entra / Google) using existing tooling
# e.g. saml2aws, granted, leapp — whichever your org uses

# Option C — IAM access key (last resort; rotate keys regularly)
aws configure             # prompts for key + secret in your terminal — never paste into chat

GCP — if you use GCP IaaS

# Check first
gcloud auth list

# If not authenticated:
gcloud auth login                              # browser flow
gcloud config set project YOUR_PROJECT_ID
gcloud auth application-default login          # for Terraform / SDKs

Firebase — if you use Firebase

# Install if needed
npm install -g firebase-tools

# Check + login
firebase projects:list
firebase login                                 # browser flow

Azure — if you use Azure

# Check + login
az account show
az login                                       # browser flow
az account set --subscription "Subscription Name or ID"

Note: Every command above runs locally. Tokens are stored in your home directory's CLI config (e.g. ~/.config/gh/, ~/.aws/sso/). They never get sent to Triarch, Claude Code chat, or any HTML rendered here.

1. Download the files

  • discovery.sh — bash script, prints metadata only (account IDs, scopes, plan tier)
  • gap-analysis.md — Claude Code prompt that scores discovery output
  • deploy.md — Claude Code prompt that walks through implementing each gap

2. Run discovery (your terminal)

chmod +x discovery.sh
./discovery.sh ORG REPO          # org and repo name; both optional
# Output saved to discovery-<timestamp>.txt — review before sharing

3. Score it (Claude Code)

# In your Claude Code session, hand it the prompt:
@gap-analysis.md
# Or just say:
"Run gap-analysis.md against the discovery-<timestamp>.txt I just produced for ORG/REPO."

Claude Code produces gap-analysis-ORG-REPO-DATE.html with the same Triarch branding as this page.

4. Deploy (Claude Code)

# Same Claude Code session — hand it the deploy prompt:
@deploy.md
# With your findings as input:
"Use deploy.md to walk through addressing the failing items in gap-analysis-ORG-REPO-DATE.html"

deploy.md never asks you to paste a token or password. When a secret is needed, it prints the gh secret set command for you to run yourself.

Credential safety guarantees

Nothing here logs, retains, or transmits credentials.

  • The form runs entirely in your browser. Answers are not sent anywhere — no analytics, no telemetry. Page reload clears them unless you download the deploy.md.
  • discovery.sh only calls gh and aws commands that print metadata (plan tier, account IDs, role names, scopes, branch protection settings). It never reads, prints, or stores tokens, secrets, passwords, or keys. Output goes to a local file you control.
  • gap-analysis.md and deploy.md are Claude Code prompts. They never instruct Claude to fetch, paste, transmit, or chat-log secret values. When a secret is required, you run gh secret set yourself in your terminal — the value never enters Claude Code's chat history.
  • The findings HTML contains no secrets. Account IDs and ARNs may appear (for context); these are not credentials. Review the file before sharing.
  • OIDC over tokens wherever cloud auth is used. The deploy prompt prefers federated identity for this exact reason — no static keys to leak.

If anything in the deploy run feels like it's asking for a credential value to paste into chat, stop and report it. That would be a deviation from the prompt.
