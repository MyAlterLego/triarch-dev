Principle 1 · Of 5

Security is a left-shifted gate,
not a right-shifted audit.

Every dangerous thing — secrets, vulnerable dependencies, unsafe code patterns, missing controls, untrustworthy build artefacts — is caught before merge or before promotion. Not in pen-test season. Not after the breach.

Stop here At PR review
→
not here At deploy time
→
definitely not here After incident
Principle 2 · Of 5

GitHub is the system of record.

Source code, infrastructure-as-code, threat models, scan history, approval evidence, release provenance — all pivot off GitHub. Audit reconstruction works from git + gh CLI alone.

Principle 3 · Of 5

Principle 4 · Of 5

Three blast-radius zones.

Each environment lives in its own cloud account, with its own credentials, its own database, and its own approval gates. Compromise of dev cannot reach prod.

dev · auto-deploy

0 reviewers

Push to main → deploys. Smoke test only. Fast feedback.

staging · gated

1 reviewer + 5 min

Manual promote. Prevent self-review. Integration tests run.

prod · hardened

2 reviewers + 30 min

Tag-only. Provenance + signature verified. Change ticket required.

Principle 5 · Of 5

Verifiable provenance.

Every container is keyless-signed via Sigstore cosign. Every release carries a SLSA build-provenance attestation. Prod refuses to deploy what it cannot verify.

Act 4 · The gates

How approval gates actually work.

No magic. GitHub Environments + protection rules + OIDC trust conditions, layered. Two pairs of eyes, a cooling-off period, and a verifiable artefact at every promotion.

push to main dev · auto-deploy · smoke test
↓ manual promote ↓
staging gate 1 reviewer + 5min wait + prevent self-review
↓ tag v*.*.* ↓
prod gate 2 reviewers + 30min wait + tag-only + change ticket + provenance check
Act 4 · What ships

Every release carries its proof.

The artefact pushed to your registry is more than a container. It's cryptographically verifiable end-to-end.

Cosign signatureKeyless via Sigstore — identity is "this workflow run, in this repo, on this ref".
SLSA build provenanceRecords source SHA, builder, build steps. Verifiable months later.
SBOM (SPDX)Every dependency, every version. Searchable when the next CVE drops.
Trivy scan resultsCritical/High vulnerabilities — SARIF — uploaded to GitHub Security tab.
Act 5 · Your choices

What's required,
what's recommended,
what's optional.

Required

Without these, no CI/CD

  • GitHub repo + Actions
  • One CI workflow
  • One deploy workflow
  • Branch protection on main
  • Deploy creds (OIDC or token)
  • A target hosting environment
Recommended

Default in this framework

  • 3 protected environments
  • OIDC instead of tokens
  • CODEOWNERS + signed commits
  • SAST + SCA + secrets scan
  • IaC for cloud setup
  • Threat model in repo
  • Audit log to immutable storage
Optional

Add when capacity allows

  • SLSA + cosign signing
  • Threat-model drift CI gate
  • DAST scanning
  • DR drill workflow
  • Cost guardrail alerts
  • License compliance check
  • Full observability stack
Act 5 · Where it runs

Pick your hosting path.

Two primary paths cover most SMB customers. Both are first-class.

⚡ Simplest path

GCP + Firebase

Hosting + auth + database in one console, ~10 min setup. Best when you have no compliance pressure and just need to ship.

10-min setup, generous free tier
Bundles hosting / auth / DB
Vendor lock-in to Google
Firebase Auth lacks SCIM (matters for enterprise B2B)
🏢 B2B / compliance path

AWS + WorkOS + Postgres

Most-trodden path for SaaS heading toward SOC 2. Three accounts, SCIM-ready auth, relational DB. The included scaffold is AWS-first.

Best B2B procurement readiness
Three blast-radius zones
Multi-day provisioning (AWS Org + 3 accounts)
More moving parts to operate

Other paths supported: Azure, full GCP IaaS, Cloudflare-only. The shape of the pipeline doesn't change.

Act 5 · Auth & DB

Authenticate users, store their data.

Both layers have a default-with-Firebase shortcut and a separate-vendors path. Pick what matches your customer's procurement story.

Auth · default

Firebase Auth

Native Google / Email / Phone / SAML / OIDC. Bundled with Firebase, zero extra setup.

Free, instant, integrates with your Firebase project
No native SCIM — blocks enterprise B2B procurement
Auth · enterprise

WorkOS or Auth0

WorkOS for B2B (SAML + SCIM + audit + passkeys, generous free tier). Auth0 for B2C and complex flows.

SCIM-ready, audit-ready, enterprise-procurement-ready
Separate vendor relationship + per-env tenants
DB · default

Firestore or Postgres (Neon)

Firestore: zero ops, scales to zero, document-store. Neon: free Postgres, branch-per-env, relational.

Both have free tiers covering SMB through MVP
Pick Firestore only if your devs are comfortable with NoSQL
