Stage 1 — Understand

2. Discovery — what exists, and what doesn't

Confirm what's actually in place before customizing anything. Commands below print metadata only — no secrets, no tokens, no credentials. "We don't have one yet" is a valid answer to most of these. Stage 2 picks defaults for gaps; Stage 4 provisions them.

What's actually required vs what's strongly recommended vs optional

Three tiers. The walkthrough below uses these badges to flag each step or sub-section.

REQUIRED Without these you don't have CI/CD

  • GitHub repo with Actions enabled
  • At least one CI workflow (lint/test/scan on PR)
  • At least one deploy workflow
  • Branch protection of some form on main
  • Deploy credentials of some form (OIDC or API token)
  • A target hosting environment

RECOMMENDED Default in this framework — skip only with reason

  • 3 protected environments (vs 1)
  • OIDC to cloud (vs long-lived tokens)
  • Required code review + CODEOWNERS
  • Required signed commits
  • SAST + SCA + secrets scanning
  • Threat model checked into repo
  • IaC for cloud setup (vs click-ops)
  • Audit log captured to immutable storage

OPTIONAL Add when capacity allows

  • SLSA build provenance + cosign signing
  • Threat-model drift gate in CI
  • DAST scanning
  • DR drill workflow
  • Cost guardrail alerts
  • License compliance check
  • Full observability stack (Grafana/Datadog)
  • Multiple SAST scanners (CodeQL + Semgrep)

Questions to ask

Walk these with the customer's eng/ops lead. Answers feed the decisions in Stage 2.

  • Source control & deployment
  • Confirmed: source code lives in GitHub. Org or just personal repos?
  • Is there an existing GitHub organization, or do we need to create one?
  • What plan is the GitHub account on? (Free / Team / Enterprise)
  • How is the app deployed today, if at all? (Manual? Vercel/Netlify free tier? Nothing?)
  • Any existing CI tool we'd need to retire? (CircleCI, Jenkins, Travis — most SMBs have nothing)
  • Cloud & infrastructure
  • Is there a cloud account today? (AWS / Azure / GCP / Cloudflare / none)
  • If yes — how many accounts / subscriptions / projects exist?
  • Where is the app actually hosted right now, if anywhere? (Vercel? localhost? a VPS?)
  • Is there any existing IaC? (Terraform / CDK / Pulumi / OpenTofu / none)
  • Is the domain registered yet? With which registrar?
  • Identity & auth
  • Is there an end-user IDP today? (WorkOS / Auth0 / Cognito / custom / none)
  • Is there a developer IDP for company logins? (Okta / Entra / Google Workspace / GitHub-only)
  • Is 2FA enforced for human accounts today?
  • Application stack
  • React variant in use? (CRA / Vite / Next.js / Remix)
  • Is there a backend yet? Language and framework, if so.
  • Is there a database today? Anything beyond a local dev DB?
  • Container, serverless, or static-only?
  • Security & compliance
  • Is there a compliance program in scope? (SOC 2 / HIPAA / PCI / GDPR / CCPA / none yet)
  • Any existing security tooling? (SAST / SCA / secrets / container scanning — most SMBs have none)
  • Are there audit log retention requirements?
  • Is there an observability stack today? (Datadog / Grafana / CloudWatch only / none)
  • Operations
  • Team size? Is there an on-call rotation?
  • Change-management ticket system and prefix? (Jira OPS-, Linear, ServiceNow, none yet)
  • Is there an incident response process or tabletop drill cadence?
  • Backup or DR posture today, if any?

Safe commands — print metadata only

Replace ORG and REPO as appropriate. None of these output tokens, keys, or passwords.

GitHub

# Auth state — shows scopes, never prints the token
gh auth status

# Your user + plan
gh api user --jq '{login, id, plan: .plan.name}'

# Orgs you can see
gh api user/orgs --jq '.[].login'

# Org info (replace ORG)
gh api orgs/ORG --jq '{login, plan:.plan.name, public_repos, total_private_repos, two_factor_requirement_enabled}'

# Top 100 repos in the org
gh repo list ORG --limit 100 --json name,visibility,isArchived,pushedAt

# Teams
gh api orgs/ORG/teams --jq '.[].slug'

# Existing branch protection on a repo
gh api repos/ORG/REPO/branches/main/protection 2>/dev/null || echo "(no protection)"

# Existing environments + rulesets + workflows
gh api repos/ORG/REPO/environments --jq '.environments[].name' 2>/dev/null
gh api repos/ORG/REPO/rulesets --jq '.[].name' 2>/dev/null
gh api repos/ORG/REPO/contents/.github/workflows --jq '.[].name' 2>/dev/null

# Org-level allowed-actions policy
gh api orgs/ORG/actions/permissions 2>/dev/null

AWS

# Current identity (account ID + ARN, no secrets)
aws sts get-caller-identity

# AWS Org structure (if accessible)
aws organizations list-accounts --query 'Accounts[*].{Id:Id,Name:Name,Status:Status}' --output table

# Existing OIDC providers
aws iam list-open-id-connect-providers

# Existing GitHub Actions roles (we don't want to collide)
aws iam list-roles --query 'Roles[?starts_with(RoleName, `github`)].{Name:RoleName,Arn:Arn}' --output table

# Default region + available regions
aws configure get region
aws ec2 describe-regions --query 'Regions[*].RegionName' --output text

Local tooling

gh      --version
git     --version
tofu    -version || terraform -version
aws     --version
docker  --version
node    --version
